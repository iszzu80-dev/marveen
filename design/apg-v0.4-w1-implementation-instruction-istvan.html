Döntési összefoglaló

Az implementáció elindítható, és a helyes első work item:

W1 — Minimal Sidecar Store
  • A pilot nem igényel Marveen core-módosítást.
  • Nincs upstream előfeltétel.
  • A tokenmegtakarítás még hipotézis, ezért a pilot feladata a mérés.
  • A sidecar teljesen eltávolítható és a Marveen core-tól elkülönített marad.

Mit kell bizonyítania a W1-nek?

1. Idempotens inicializálás

első init → létrehozza a DB-t
második init → tiszta no-op
megszakított init → helyreállítható

2. Verziózott migrációk

  • schema_version
  • migration_id
  • applied_at
  • checksum

A módosított, már lefuttatott migrációt vissza kell utasítani.

3. Append-only kanonikus rekordok

  • artifact version;
  • edge;
  • execution receipt;
  • evidence reference;
  • checkpoint result;
  • token measurement;
  • profile activation;
  • replay run event.

4. Idempotency

Ugyanazon logikai művelet ismétlése ne hozzon létre duplikátumot.

5. Lineage head külön kezelése

lineage_id → current artifact version

A régi verziók megmaradnak.

6. Biztonságos SQLite-konfiguráció

  • foreign_keys=ON
  • WAL mód
  • busy_timeout
  • explicit tranzakciók
  • paraméterezett SQL
  • explicit indexek
  • UTC timestamp

7. Adatminimalizálás

A store ne tartalmazzon API-kulcsot, session tokent, teljes JWT-t, nyers promptot, session historyt, valódi user emailt, nyers deployment-logot vagy nagy evidence payloadot.

Marveennek visszamásolható implementációs utasítás

Az APG v0.4 Pilot Kernel implementációs specifikációját elfogadom. Kezdd el kizárólag a W1 — Minimal Sidecar Store implementációját.

FONTOS HELYKORREKCIÓ

A sidecar ne kerüljön az official Marveen repositoryba, még gitignored könyvtárként sem.

Célhely: ~/marveen-local/apg-kernel/

~/marveen-local/apg-kernel/
├── src/
├── migrations/
├── tests/
├── README.md
└── store/
    └── apg-kernel.db

A schema és a migrációk a sidecar saját verziózott source-ában legyenek. Az adatbázis legyen gitignored. A Marveen core repository és core DB nem módosítható.

SCOPE

Csak W1-et implementáld. Ne kezdj bele W2–W9-be, még akkor sem, ha valamelyik könnyen hozzáadhatónak tűnik.

A W1 célja kizárólag a minimális, megbízható sidecar persistence foundation.

KÖTELEZŐ TÁROLÁSI TERÜLETEK

  • canonical artifact versionök;
  • lineage headek;
  • edge-ek;
  • execution receiptek;
  • evidence reference-ek;
  • checkpoint resultok;
  • token measurement recordok;
  • profile activation recordok;
  • replay runok és replay eventek;
  • schema migration history.

A tároló most csak generikus recordokat kezeljen. Ne implementálja a checkpoint-, token-, identity- vagy evidence-chain üzleti logikáját.

KÖTELEZŐ TULAJDONSÁGOK

1. Idempotens init

  • első futás létrehozza a DB-t;
  • ismételt futás tiszta no-op;
  • részlegesen sikertelen init ne hagyjon használhatatlan állapotot.

2. Verziózott migráció

  • migration ID;
  • checksum;
  • applied_at;
  • schema version;
  • már alkalmazott migráció tartalmának megváltoztatása hibát adjon.

3. Append-only szemantika

Az artifact version, edge, execution receipt, evidence reference, checkpoint result, token measurement, profile activation és replay event ne legyen helyben módosítható.

4. Idempotency

  • ugyanazon logikai write ismétlése ne hozzon létre duplikátumot;
  • az idempotency key ne tartalmazzon attempt számot, ha ugyanarról a logikai műveletről van szó.

5. Lineage

  • immutable artifact versionök;
  • külön lineage-head projection;
  • a lineage head csak létező, azonos lineage-hez tartozó artifact versionre mutathat.

6. SQLite safety

  • foreign_keys=ON;
  • WAL;
  • busy_timeout;
  • explicit transaction;
  • paraméterezett SQL;
  • szükséges indexek;
  • UTC timestamp.

7. Adatminimalizálás

A store nem tárolhat credentialt, API-kulcsot, session tokent, teljes JWT-t, nyers promptot vagy session historyt, valódi user emailt vagy más szükségtelen PII-t, nyers deployment-logot vagy nagy nyers evidence payloadot.

Csak digest, locator, külső referencia és strukturált metadata tárolható.

KÖTELEZŐ TESZTEK

  • clean database init;
  • repeated init;
  • migration ordering;
  • migration checksum mismatch rejection;
  • valid artifact append;
  • duplicate idempotent append;
  • immutable artifact mutation rejection;
  • lineage-head advancement;
  • invalid lineage-head rejection;
  • legal edge insertion;
  • illegal edge rejection;
  • missing edge target rejection;
  • transaction rollback;
  • database close/reopen/readback;
  • concurrent write vagy lock-contention viselkedés;
  • secret/PII negative fixture rejection;
  • clean removal után a Marveen core változatlanul működik.

OUT OF SCOPE

  • API server;
  • dashboard;
  • daemon/service;
  • Lifecycle Runner;
  • live agent dispatch;
  • Kanban write-back;
  • Marveen core DB adapter;
  • provider adapter;
  • token-origin parsing;
  • test-identity propagation;
  • checkpoint evaluator;
  • profile business logic;
  • v0.5 Ralph loop;
  • frontend/backend delivery lane;
  • általános graph database;
  • fölösleges ORM vagy framework.

IMPLEMENTÁCIÓS FEGYELEM

  • A jóváhagyott apg-v0.4-pilot-kernel-store-schema.sql legyen az alap.
  • Ettől csak igazolt hibajavítás vagy SQLite-kompatibilitási ok miatt térj el.
  • Minden eltérést dokumentálj.
  • Ne növeld a táblák vagy mezők számát konkrét fogyasztó nélkül.
  • Minden tábla mellett dokumentáld, mely későbbi P0 komponens használja.
  • A W1 végén futtasd az összes determinisztikus tesztet.
  • Ne implementálj párhuzamosan más work itemet.
  • Ne módosíts Marveen core source-t, core DB-t, CLAUDE.md-t, skillt, schedulert vagy servicet.
  • Ne deployolj.
  • Ne nyiss upstream issue-t vagy PR-t.
